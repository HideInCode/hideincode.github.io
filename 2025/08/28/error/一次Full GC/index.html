一次 Full GC 频繁的排查过程

2025-08-28 问题排查 error 0 评论

上周下午，监控突然开始告警，某个核心服务的接口响应时间从几十毫秒飙升到几秒，上游已经开始超时重试。看了下 Grafana，JVM 的 Full GC 频率从之前的几小时一次变成了几分钟一次，每次停顿接近两秒，典型的“服务要挂”味道。

一、先看现象，稳住局面

先确认是不是个别实例的问题。登录到其中一台机器，用 top 看了下，Java 进程 CPU 上蹿下跳，内存占用几乎打满 Xmx。再看 GC 情况：

jstat -gc <pid> 1000 10

FGC 那一列涨得飞快，FGCT 不断增加，E 和 O 区几乎都是满的，老年代回收后几乎没释放多少，说明老年代里的东西都是“活的”，没法回收，但新生代也源源不断有对象晋升。

这时候还不能重启，重启什么证据都没了。先保留现场：

jmap -dump:live,format=b,file=/tmp/heap.bin <pid>

同时顺手把线程栈也抓了：

jstack <pid> > /tmp/jstack.log

二、看线程，先排除掉一些简单可能

jstack 里没看到死锁，但发现大量线程卡在同一个数据库查询上。看了下数据库连接池，连接被打满，部分线程在等待获取连接。这说明数据库压力巨大，调用链上肯定有慢查询。

不过慢查询不一定是根因，也可能是结果。因为内存快被撑爆，GC 频繁，CPU 被 GC 线程吃满，整体吞吐下降，大量请求堆积，连锁反应把数据库也拖慢了。

所以重点还是回到堆里，看看到底是什么东西占着老年代不放。

三、分析堆转储，找到真凶

heap.bin 有 4 个多 G，本地 IDEA 加载不了，直接上 Eclipse MAT。打开后，Leak Suspects 报告立刻指向一个非常明显的对象：

com.mysql.cj.jdbc.result.ResultSetImpl 的实例占用了接近 2 GB 内存。

展开看内部，每个 ResultSetImpl 里都挂着一个巨大的 byte[]，再看引用链，这些 ResultSet 都来自同一个 Service 方法，方法里有一个查询订单明细的 SQL，但没带任何分页条件，全表扫了，而且数据还在不断增长。

进一步看 GC Roots，这些 ResultSet 被 ThreadLocal 引着。原来代码里用了一个分页工具，把查询结果集临时放到了 ThreadLocal 里，但忘记清理了。Tomcat 线程是池化的，线程复用后 ThreadLocal 还挂着上一次请求的大对象，导致这些对象永远不能被回收。

四、验证结论

回到代码里确认，果然在那个 Service 方法里找到了这段：

private ThreadLocal<List<OrderDetail>> detailHolder = new ThreadLocal<>();

public void process() {
    List<OrderDetail> details = orderDetailMapper.selectAll(); // 无分页
    detailHolder.set(details);
    // 使用 detailHolder...
    // 最后没有 remove()
}

每次请求进来，selectAll() 查出几十万条记录，塞进 ThreadLocal，请求结束线程回收，但 ThreadLocal 的值还在，下一次复用时又被覆盖，但旧值依然被线程对象引用，GC 无法回收，慢慢把老年代撑爆。

这也解释了为什么 jstat 里看到老年代一直很高——全是堆在那里的 ResultSet 和 byte[]。

五、修复和效果

修复很简单：

  1. 查询加上分页，避免一次拉全表。
  2. 在 finally 块里调用 detailHolder.remove()。
  3. 同时把 ThreadLocal 改为局部变量，因为这里根本没有跨方法传递的必要，直接去掉 ThreadLocal 更好。

修完后上线，老年代从 4G 降到 600M，Full GC 频率回到几小时一次，接口响应时间恢复到几十毫秒。

六、后续要做的

  • 把 GC 日志加上，-Xlog:gc*:file=/path/gc.log:time,uptime:filecount=10,filesize=100M，避免下次再临时抓瞎。
  • 给 ThreadLocal 使用加了全局代码审查规则，只要看到 ThreadLocal.set，必须检查是否有对应的 remove。
  • 数据库慢查询也一并加了索引，毕竟之前虽然不拉全表，但现在分页后也要保证查询效率。

整个排查过程花了不到半小时，工具只是 jstat、jmap、MAT，思路也很清晰：先确认现象、保留现场，然后从线程栈和堆转储两条线同时排查，最终锁定 ThreadLocal 忘记清理导致的内存泄漏。这种问题在线上其实挺常见的，记录下来权当一个警示。
