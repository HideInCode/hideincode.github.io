常见问题排查思路

一、排查思路框架

遇到线上问题不要慌，按以下顺序逐步缩小范围：

  1. 明确现象：用户反馈什么？页面报错？接口超时？服务假死？
  2. 确认影响面：是个别用户还是全部？单个实例还是整个集群？
  3. 最近变更：是否上线了新代码、修改了配置、有流量波动？
  4. 系统指标：CPU、内存、磁盘、网络是否异常？
  5. 应用指标：慢 SQL？线程池满？GC 频繁？连接池耗尽？
  6. 日志与跟踪：错误日志、调用链是否有线索？
  7. 复现与验证：能否在测试环境重现？临时解决方案是什么？

原则：先止损，再查根因。 必要时先重启、回滚或限流，保留现场数据（dump、日志）再恢复。

二、系统级排查命令

2.1 CPU 与负载

# 实时查看 CPU、内存、负载
top

# 更详细的 CPU 信息，按 1 展开每个核心
mpstat -P ALL 1

# 查看系统负载历史（1、5、15分钟平均值）
uptime

关注点：CPU 利用率（us 用户态、sy 内核态、wa IO等待），负载值超过 CPU 核心数通常表示存在排队。

2.2 内存

# 查看物理内存和 swap 使用概况
free -h

# 查看内存详细统计（含缓存、缓冲区）
cat /proc/meminfo

# 按内存使用排序的进程列表
ps aux --sort=-%mem | head -20

常见问题：available 内存很低，swap 使用持续增长，可能存在内存泄漏或内存不足。

2.3 磁盘与 IO

# 查看磁盘使用率
df -h

# 查看目录大小
du -sh /var/log/*

# 查看磁盘 IO 情况
iostat -x 1

# 查看哪些进程在大量读写磁盘
iotop -o

关注点：/ 或 /var 分区满了会导致服务不可用，iostat 的 %util 接近 100% 表示磁盘 IO 瓶颈。

2.4 网络

# 查看监听端口及服务
netstat -tlnp
# 或使用 ss（更高效）
ss -tlnp

# 查看网络连接状态统计
ss -s

# 查看某个端口连接数
ss -ant | grep :80 | wc -l

# 实时查看网络流量
nethogs

# 抓包分析
tcpdump -i eth0 port 8080 -w /tmp/capture.pcap

TIME_WAIT 过多可能影响连接建立，SYN_RECV 堆积可能是 SYN Flood 攻击或应用响应慢。

三、Java 应用诊断命令

3.1 线程与线程栈

# 查看 Java 进程
jps -l

# 打印所有线程的堆栈信息（类似 jstack）
jstack <pid>

# 查看线程 CPU 消耗（按线程）
top -Hp <pid>
# 将线程 ID 转为 16 进制，然后在 jstack 输出中搜索

# 快速查看死锁
jstack <pid> | grep -A 30 "deadlock"

适用场景：接口慢、线程死锁、线程池满。

3.2 堆内存与 GC

# 查看堆内存使用和 GC 概况
jstat -gc <pid> 1000 10

# 查看堆直方图（占用内存最多的类）
jmap -histo <pid> | head -50

# 生成堆转储文件（会触发 Full GC，谨慎）
jmap -dump:live,format=b,file=/tmp/heap.bin <pid>

# 查看 GC 日志（需提前配置）
# -Xlog:gc*:file=/path/to/gc.log:time,uptime:filecount=10,filesize=100M

GC 频繁：jstat -gc 中 FGC 持续增长，E 和 O 区长时间高占比，需要调优或排查内存泄漏。

3.3 参数与配置

# 查看 JVM 启动参数
jinfo -flags <pid>

# 查看某个 JVM 系统属性值
jinfo -sysprops <pid> | grep file.encoding

用于检查 Xmx、Xms、GC 策略等是否与预期一致。

四、应用日志与请求追踪

4.1 日志定位

# 实时查看应用日志，过滤 ERROR
tail -f app.log | grep ERROR

# 查看最近 1000 行
tail -n 1000 app.log

# 按时间段查找（如 10:00-10:05）
sed -n '/2026-05-26 10:00/,/2026-05-26 10:05/p' app.log

# 统计某个错误出现次数
grep "NullPointerException" app.log | wc -l

技巧：把错误栈的关键信息复制到编辑器，搜索相似问题；配置日志采样率避免磁盘打满。

4.2 接口调用排查

# 快速检查接口返回
curl -v http://localhost:8080/api/health

# 查看接口耗时
curl -w "\n%{time_total}s\n" http://localhost:8080/api/info

# 测试端口是否通
telnet localhost 8080
nc -zv localhost 8080

# 跟踪路由
traceroute <目标IP>

五、容器化环境排查

5.1 Docker 环境

# 查看容器日志
docker logs --tail 100 <容器ID>

# 进入容器内部
docker exec -it <容器ID> /bin/sh

# 查看容器资源使用
docker stats

5.2 Kubernetes 环境

# 查看 Pod 状态和事件
kubectl get pods -n <namespace>
kubectl describe pod <pod-name> -n <namespace>

# 查看 Pod 日志（多容器需加 -c）
kubectl logs -f <pod-name> -n <namespace> --tail=200

# 进入容器
kubectl exec -it <pod-name> -n <namespace> -- /bin/sh

# 查看资源限制
kubectl describe node <node-name>

六、常见问题速查

现象 可能原因 排查命令/方法
接口响应慢 慢 SQL、线程池满、下游超时 jstack、慢 SQL 日志、APM 追踪
CPU 飙升 死循环、大量 GC、加密解密 top -Hp + jstack 定位线程
内存溢出（OOM） 内存泄漏、堆设置过小 jmap -dump，使用 MAT 分析
磁盘空间满 日志未轮转、临时文件堆积 `du -sh /*
连接数满/端口不够用 TIME_WAIT 过多、连接未释放 ss -s、netstat，调整内核参数
应用假死（无响应） 死锁、线程全部阻塞、OOM jstack 查看线程状态，jstat -gc

七、总结经验

  • 监控先行：CPU、内存、磁盘、网络、GC、线程池、接口耗时等关键指标必须接入监控告警。
  • 保留现场：优先保留日志、dump、抓包文件，重启会丢失这些信息。
  • 统一诊断入口：对 Spring Boot 应用集成 Actuator + Spring Boot Admin 或 Arthas，可在线诊断。
  • 预案演练：常见故障应有标准操作流程（SOP），减少恢复时间。

排查就像医生看病，望闻问切，缺一不可。熟练使用这些命令和思路，你就能在复杂线上环境中快速破案。
