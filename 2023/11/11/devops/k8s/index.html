K8S集群搭建

对中小团队或个人开发者而言，使用 kubectl 命令行管理 K8s 集群，宛如“驾驶一辆没有仪表盘的汽车——你能开动它，却难以感知状态，更无法精细操控”。这时，一款轻量、易用的可视化管理面板能极大降低运维门槛。Kuboard 正是为此而生。

本文将带你走完一条完整的实践链路：从使用 kubeadm 搭建 Kubernetes 集群、配置容器运行时与网络插件，到部署 Kuboard v3 并通过其图形界面完成应用发布与监控。全文基于 Ubuntu 22.04 系统演示，兼容主流 Linux 发行版。

1. 环境准备

1.1 硬件与系统

  • 操作系统：推荐 Ubuntu 20.04/22.04 或 CentOS 7/8
  • 节点规划：至少 1 个控制平面 (master) + 1 个工作节点 (worker)
  • 配置要求：每台机器 ≥ 2 核 CPU、4 GB 内存、50 GB 磁盘

7. Kuboard 核心功能实操

7.1 图形化部署应用

  1. 切换到 “default” 命名空间（或新建一个）。
  2. 点击 “创建工作负载” → 选择 “Deployment”。
  3. 填写服务名称（如 nginx-demo），添加工作容器，指定镜像 nginx:1.23-alpine，设置容器端口 80。
  4. 点击“保存”，Kuboard 会自动生成并应用 YAML，无需手动编写。

7.2 实时日志与终端

  • 查看日志：在 Deployment 详情页，点击任意 Pod → “日志” 标签，支持按容器筛选和实时刷新，无需输入 kubectl logs -f。
  • 容器终端：点击 Pod 详情页的 “终端” 按钮，直接进入容器内部执行命令，相当于 kubectl exec -it。

7.3 资源监控

Kuboard 内置了资源层监控套件。安装后，可直观查看节点和 Pod 的 CPU、内存等指标，支持阈值告警（如 Pod 内存超 80% 高亮提示）。

8. 常见问题排查

以下是根据实战经验总结的高频问题与解决思路：

  • **Pod 无法启动 / 镜像拉取失败 (ErrImagePull)**：首先 docker pull 或在各节点上手动拉取镜像确认网络可达；若为私有仓库，需在 Kuboard 创建镜像仓库密文并关联到 ServiceAccount。
  • Kuboard 登录后加载不出集群列表：通常是 token 过期导致。可用 kubeadm token create --ttl 0 生成永不过期的 token，重新导入配置。
  • 导入集群时提示 “timeout”：通常因 Kuboard 无法访问 K8s API Server 所致。检查防火墙规则，确保 Kuboard 所在服务器能 ping 通 kubernetes.default.svc，且 master 节点端口 6443 已放行。
  • **Kuboard 组件 CrashLoopBackOff**：先确认 master 节点已打上标签 kubectl label nodes <master-node-name> k8s.kuboard.cn/role=etcd；未打标会导致 kuboard-etcd 无法调度。
  • 容器中 docker ps 失效：K8s 接管后，应改用 crictl ps (containerd) 或 kubectl get pods 查看容器，直接操作 Docker 会造成 Pod 状态异常。

9. 结语

至此，你已拥有一套从底层集群搭建到上层可视化管理覆盖的完整技术栈。回顾整个过程，核心要点可归纳为：

  1. 容器运行时的选择决定后续运维复杂度，containerd 是当前主流方向。
  2. 网络插件是集群从“摆设”到“可用”的关键一环，Flannel 适合快速上手，Calico 适合生产。
  3. Kuboard 将复杂的 K8s 管理转化为直观操作，让开发人员无需记忆海量命令，让运维人员通过仪表盘快速发现异常。

希望本文能帮助你少走弯路，快速建立起 K8s 管理与监控的实战能力。后续我会继续分享如何在这套体系上集成 CI/CD 流水线与日志聚合方案，欢迎持续关注。
