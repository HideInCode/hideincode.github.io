ES的使用与原理

2020-06-11 系统设计 system design 0 评论

ElasticSearch

Elasticsearch 是一个基于 Lucene 构建的分布式、RESTful 风格的搜索和分析引擎。它能够快速存储、搜索和分析海量数据，并支持近实时查询。

1.2 核心概念对比（与关系型数据库）

Elasticsearch 关系型数据库 (MySQL)
Index (索引) Database (数据库)
Type (已废弃) Table (表)
Document (文档) Row (行)
Field (字段) Column (列)
Mapping (映射) Schema (表结构)
DSL (查询语言) SQL

注意：从 7.0 开始，一个索引只能有一个类型（_doc），类型概念正逐步移除。

1.3 倒排索引原理

ES 的核心是倒排索引。它把每个文档中的词拆分成词条（Term），建立“词条 → 文档ID”的映射，从而实现全文搜索的极致速度。

// 原始文档
Doc1: "Elasticsearch 快速入门"
Doc2: "快速掌握 Elasticsearch"
Doc3: "搜索引擎对比"

// 倒排索引（简化）
快速      → [Doc1, Doc2]
Elasticsearch → [Doc1, Doc2]
入门      → [Doc1]
掌握      → [Doc2]
搜索      → [Doc3]
引擎      → [Doc3]
对比      → [Doc3]

二、索引与映射

2.1 索引创建

PUT /my_index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
      "title": { "type": "text", "analyzer": "ik_max_word" },
      "price": { "type": "float" },
      "create_time": { "type": "date", "format": "yyyy-MM-dd HH:mm:ss" }
    }
  }
}

分片数创建后不可修改，需合理规划；副本数可动态调整。

2.2 字段类型

  • text：全文搜索，会被分词
  • keyword：精确匹配，不分词，适合聚合、排序
  • date：日期类型，可指定格式
  • long / integer / float / double：数值类型
  • boolean：布尔类型
  • object / nested：复杂结构，nested 保留内部文档边界，可独立查询

动态映射：未预先定义的字段，ES 会自动推测类型，可能引入问题（如 ID 被映射为 text）。生产环境建议关闭动态映射或使用严格模板。

三、查询 DSL

3.1 全文搜索

  • match：对查询串进行分词后执行全文匹配

  • match_phrase：短语匹配，要求词连续且顺序一致

  • multi_match：在多个字段上搜索

    GET /my_index/_search
    {
    “query”: {
    “multi_match”: {
    “query”: “Elasticsearch 入门”,
    “fields”: [“title”, “description”]
    }
    }
    }

3.2 精确匹配

  • term：按词条精确查询（keyword 字段）
  • terms：多值精确匹配，相当于 SQL 的 IN
  • range：范围查询，支持 gt、gte、lt、lte

3.3 组合查询

  • bool 查询：组合 must、filter、should、must_not
    • must：必须匹配，贡献得分
    • filter：过滤，不贡献得分（缓存）
    • should：或逻辑，最小匹配数可配置

3.4 聚合分析

  • bucket：分组（如 terms、date_histogram）

  • metric：统计（如 avg、sum、min、max、stats）

  • pipeline：对聚合结果再聚合

    GET /orders/_search
    {
    “size”: 0,
    “aggs”: {
    “by_category”: {
    “terms”: { “field”: “category.keyword” },
    “aggs”: {
    “avg_price”: { “avg”: { “field”: “price” } }
    }
    }
    }
    }

四、集群架构与数据路由

4.1 节点角色

  • Master 节点：负责集群管理（创建/删除索引、节点健康监测）
  • Data 节点：存储数据，执行 CRUD、搜索、聚合
  • Ingest 节点：预处理管道，在索引前转换数据
  • Coordinating 节点：协调请求分发，合并结果（默认所有节点都是协调节点）

4.2 分片与副本

一个索引被分成多个分片（Primary Shard），每个分片可以有多个副本（Replica Shard）。副本分片既提升查询吞吐量，又保证高可用。

路由公式：

shard_num = hash(document_id) % num_primary_shards

因此分片数固定后不能修改（除非 reindex）。

4.3 健康状态

  • green：所有主分片和副本分片均已分配
  • yellow：主分片全部可用，但部分副本未分配（可能是单节点）
  • red：部分主分片不可用，数据缺失

五、性能优化实践

5.1 批量写入优化

  • 使用 _bulk API 批量索引，单次建议 5~15MB
  • 适当增大 refresh_interval（默认 1s），写入时可设为 -1 暂停刷新，完成后再恢复
  • 减少副本数（如设为 0）加快写入，写入后恢复

5.2 查询优化

  • 避免深度分页：from + size 过大会深度遍历，使用 search_after 或滚动（Scroll）API
  • filter 优于 must：filter 结果可缓存，减少计算
  • 避免通配符开头：*word 会全表扫描
  • 合理使用 routing：指定路由值将查询限定在单个分片

5.3 内存与存储

  • JVM 堆内存：不超过机器内存的 50%，最高 30GB（指针压缩阈值）
  • 禁用 swap：避免内存交换导致性能急剧下降
  • 使用 SSD：尤其频繁写入和随机读的场景

5.4 索引设计

  • 冷热分离：按时间滚动的索引，将旧索引迁移到廉价存储节点
  • 数据精简：合理选择字段类型，关闭不必要的 index 或 source
  • 动态 mapping 严格化：避免意外字段类型导致映射爆炸

六、常见问题排查

6.1 集群状态 yellow / red

  • yellow：通常由于副本无法分配，常见原因：节点不足、磁盘空间超过 95%（read_only_allow_delete 被触发）
  • red：主分片丢失，通常由于节点长时间离线或数据损坏

处理方法：_cat/shards?v 查看未分配的分片，_cluster/allocation/explain 分析原因。

6.2 搜索慢

  • 使用 Profile API 查看查询耗时分布
  • 检查查询是否用了 leading wildcard 或低效正则
  • 查看节点 CPU 和 GC 频率，可能需要扩容或优化查询

6.3 写入慢

  • 检查磁盘 IO，普通 HDD 随机写性能差
  • refresh_interval 太短导致频繁生成段文件
  • 聚合分析过多消耗 CPU

6.4 内存溢出（OOM）

  • 堆内存设置过大（>31GB）或过小
  • fielddata 用于 text 字段聚合（默认关闭），开启后需谨慎
  • scroll 查询未及时清除，占用大量内存

6.5 脑裂（Split Brain）

经典原因：网络分区导致多个 Master 产生，破坏数据一致性。
预防配置：discovery.zen.minimum_master_nodes 设置为 (master候选节点数 / 2) + 1，避免少数派选举。
（注意：ES 7.0 以上已使用新的基于八卦协议的集群发现机制，但原理相近。）

七、与 Spring Boot 集成要点

7.1 Spring Data Elasticsearch

使用 ElasticsearchRestTemplate 或 ElasticsearchRepository 简化操作。

@Document(indexName = "products")
public class Product {
    @Id
    private String id;
    @Field(type = FieldType.Text, analyzer = "ik_max_word")
    private String name;
    // ...
}

public interface ProductRepository extends ElasticsearchRepository<Product, String> {
    List<Product> findByName(String name);
}

7.2 客户端版本匹配

确保 Spring Data ES 与 ES 服务端版本兼容，否则可能出现序列化或 API 调用错误。

八、面试高频问答

问：ES 为什么快？
答：倒排索引结构，基于内存和文件系统缓存加速，分片并发查询，无 JOIN 等复杂关联。

问：如何保证数据一致性？
ES 是近实时（near real-time），默认 1 秒 refresh。可通过 refresh=wait_for 强读刷新，或使用外部强一致存储做数据源。

问：分词器是什么？常用有哪些？
分词器将文本分解为词条。常用：standard（英文默认）、ik_smart / ik_max_word（中文）、pinyin 等。

问：如何设计日志收集架构？
Filebeat → Kafka → Logstash → Elasticsearch → Kibana（经典的 ELK 栈），利用 Kafka 做缓冲解耦。

熟练掌握这些基础和实践，你就能够应对日常 ES 的开发与面试挑战。
