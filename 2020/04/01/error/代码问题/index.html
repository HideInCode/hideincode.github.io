代码问题排查思路

一、排查前的准备：三问法

遇到 Bug 不要立刻看代码，先问自己三个问题：

  1. 现象是什么？ 报错信息？数据不对？响应慢？功能无响应？
  2. 影响范围多大？ 所有用户还是特定用户？所有环境还是个别环境？最近一次正常是什么时候？
  3. 最近变更了什么？ 代码提交、配置修改、依赖升级、数据迁移？多数故障源于变更。

原则：先稳定现场（保留日志、堆转储），再尝试复现，最后才是修复。

二、逻辑错误的排查

逻辑错误通常不会直接报错，但结果与预期不符。

2.1 数据流追踪法

从入口开始，用实际请求参数逐步跟踪代码执行路径，打印或记录每一步的关键变量值。

// 示例：一个计算方法没有返回预期值
public int calculate(int a, int b) {
    int result = a + b;
    // 怀疑这里逻辑有误，插入临时日志
    System.out.println("a=" + a + ", b=" + b + ", result=" + result);
    if (result > 100) {
        result = result * 2; // 预期之外的行为？
    }
    return result;
}

更优雅的方式：使用条件断点或日志框架，避免临时 println 污染代码。

2.2 分支覆盖审查

画出方法内的条件判断流程图，用测试用例覆盖每个分支。特别留意：

  • if-else 中条件写反（如 > 写成 <）
  • 循环边界错误（< 与 <=）
  • 空集合、空字符串、null 值的边界情况

2.3 单元测试反证法

快速编写一个会失败的单元测试，证明 Bug 的存在，再通过修改代码让测试通过。这能有效防止回归。

三、异常问题的排查

3.1 异常栈信息解读

从下往上读堆栈：最底部的 Caused by 才是根因。

java.lang.RuntimeException: 订单创建失败
    at com.example.service.OrderService.create(OrderService.java:30)
    ...
Caused by: java.sql.SQLIntegrityConstraintViolationException: Duplicate entry '12345' for key 'PRIMARY'
    at com.mysql.cj.jdbc.exceptions.SQLError.createSQLException(SQLError.java:117)
    ...

上面的例子表明，RuntimeException 是业务包装后的异常，真正的根因是主键冲突。

3.2 静默吞异常

检查代码中是否有 try-catch 后只打日志不处理、甚至空 catch 块。

// 危险的写法
try {
    process();
} catch (Exception e) {
    // 什么都不做，或只打一行日志，导致后续逻辑在错误状态下继续
    log.error("处理失败", e);
}

排查方法：在 IDE 中全局搜索 catch (，审查每处异常处理逻辑是否合理。引入全局异常拦截器时，要确保不会误吞异常。

3.3 资源未关闭

数据库连接、文件流、网络连接未正确关闭，最终导致资源耗尽。

查找模式：搜索 new FileInputStream、new BufferedReader 等，检查是否在 finally 块或 try-with-resources 中关闭。

四、性能问题的排查

4.1 慢在哪里？——分阶段计时

用最简单的方式确定耗时瓶颈：

long start = System.currentTimeMillis();
// 业务逻辑 A
long afterA = System.currentTimeMillis();
// 业务逻辑 B
long afterB = System.currentTimeMillis();
// 业务逻辑 C
long afterC = System.currentTimeMillis();

log.info("A耗时:{}ms, B耗时:{}ms, C耗时:{}ms", afterA-start, afterB-afterA, afterC-afterB);

用 StopWatch 工具类（Spring 自带 StopWatch）更整洁。

4.2 数据库慢查询

ORM 框架通常会打印 SQL 日志。关注：

  • 是否没有用索引（EXPLAIN 查看执行计划）
  • 是否 N+1 查询（循环内发起子查询）
  • 是否返回过多无用字段和行数

MyBatis 开启日志：设置 logging.level.包名.mapper=debug 查看具体 SQL 和参数。

4.3 循环内的重操作

// 坏味道：循环中调用 RPC 或数据库
for (String id : ids) {
    Account account = accountService.getById(id); // 远程调用
    // ...
}

优化方向：批量查询接口、缓存、异步处理。

4.4 使用 Profiler 定位

工具推荐：

  • JProfiler、YourKit：商业级 CPU 和内存分析

  • VisualVM：免费，基本够用

  • Arthas：阿里巴巴开源的线上诊断工具，可通过命令查看热点方法

    Arthas 追踪某个类的方法执行耗时

    trace com.example.service.OrderService processOrder

五、并发问题的排查

并发 Bug 难以复现，但代码审查有迹可循。

5.1 共享变量安全

检查所有类级别或静态变量，多线程访问时是否加了锁或使用了并发容器。

// 线程不安全的计数器
private int count = 0;
public void increment() {
    count++; // 非原子
}

// 修正
private AtomicInteger count = new AtomicInteger(0);

5.2 锁的使用审计

  • 锁范围是否过大导致性能下降？是否过小导致保护不全？
  • 锁对象是否正确（避免用非 final 对象或基本类型包装类）？
  • 是否存在锁嵌套（可能导致死锁）？

死锁排查：用 jstack 或 Arthas 的 thread -b 找出死锁线程。

5.3 线程池误用

  • 是否使用了无界队列导致内存溢出？
  • 任务执行时间过长是否撑满线程池？
  • 是否在任务内部又向同一个线程池提交任务（可能死锁）？

代码审查时关注 ThreadPoolExecutor 的构造参数和 submit 的使用位置。

六、环境与配置问题的排查

6.1 环境差异

“我这能跑啊”往往是因为环境差异。比对：

  • JDK 版本、JVM 参数
  • 数据库版本、表结构、数据量
  • 第三方服务可达性（网络策略）
  • 操作系统、时区、文件编码

快速检查：打包测试环境的全量配置，与线上比对。

6.2 配置加载顺序

Spring 应用中，配置可能存在于多个位置：application.properties、环境变量、命令行参数、配置中心。优先级会覆盖。

# 查看 Spring 环境配置
curl http://localhost:8080/actuator/env

定位哪个配置最终生效，是否符合预期。

6.3 依赖冲突

Maven/Gradle 传递性依赖可能导致类方法找不到（NoSuchMethodError）、版本不兼容。

# 查看依赖树
mvn dependency:tree -Dincludes=com.fasterxml.jackson.core

IDE 中也可通过依赖分析插件快速查看冲突。

七、使用工具加速排查

场景 工具 / 方法
远程诊断 Arthas (watch, trace, monitor)
堆内存分析 Eclipse MAT, JProfiler
线程分析 jstack, fastThread
接口调用链跟踪 SkyWalking, Jaeger, Zipkin
日志聚合搜索 ELK (Elasticsearch, Logstash, Kibana)
本地调试 IDE 断点、条件断点、表达式求值
代码审查 SonarQube, 人工走查
压力测试 JMeter, wrk, Apache Bench

八、排查心法总结

  1. 分层定位：从表现层到业务层到数据层，层层剥茧。
  2. 先怀疑自身代码：不要一开始就怀疑框架或 JVM 的 Bug，99% 的问题是自己导致的。
  3. 记录排查过程：在 issue 或文档中写下每一步尝试和结果，避免重复工作。
  4. 完善监控和日志：好日志能让你秒级定位，否则排查基本靠猜。
  5. 复盘与沉淀：解决后写复盘报告，升级系统防护，防止同类问题再次发生。

排查就像破案，线索往往藏在细节里。掌握这套方法论，你将不再惧怕线上问题，而是享受“破案”的成就感。
