Dubbo

RPC框架，用于服务之间调用。

一、基础概念

1.1 Dubbo 是什么？

Dubbo 是一款由阿里巴巴开源的高性能、轻量级 Java RPC 框架，提供了服务自动注册与发现、负载均衡、容错、动态配置等面向微服务的治理能力。当前最新版本已进入 Apache 顶级项目（Apache Dubbo）。

1.2 核心能力

  • 远程通信：透明化的 RPC 调用，像调用本地方法一样调用远程服务
  • 服务治理：注册中心管理服务地址、路由规则、配置中心动态下发配置
  • 容错与扩展：多种集群容错策略，支持 SPI 扩展点机制

二、核心架构与组件

Dubbo 架构分为四大角色和三大中心：

┌─────────────┐          ┌─────────────┐
│  Provider   │          │  Consumer   │
│  (服务提供方) │          │  (服务消费方) │
└──────┬──────┘          └──────┬──────┘
       │                        │
       └───────┬────────────────┘
               │
        ┌──────▼──────┐
        │   Registry  │
        │  (注册中心)   │
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │   Monitor   │
        │  (监控中心)   │
        └─────────────┘

核心组件：

  • Provider：暴露服务的提供方
  • Consumer：调用远程服务的消费方
  • Registry：服务注册与发现中心（如 Nacos、Zookeeper、Redis）
  • Monitor：统计调用次数、耗时等监控信息
  • Container：服务运行的容器（如 Spring 容器）

工作流程：

  1. Provider 启动时向 Registry 注册自己的服务
  2. Consumer 启动时向 Registry 订阅所需的服务，Registry 推送服务地址列表
  3. Consumer 基于负载均衡策略选择一个 Provider 发起直接调用
  4. 双方定时向 Monitor 上报调用统计

三、RPC 调用原理

Dubbo 的一次 RPC 调用由代理层、协议层、传输层、序列化层等多层协同完成。

// 服务接口
public interface GreetingService {
    String sayHello(String name);
}

// Consumer 侧调用（看起来像本地调用）
@Reference
private GreetingService greetingService;

public void test() {
    String result = greetingService.sayHello("world");
}

底层调用链：

  1. Proxy：通过 JDK 或 javassist 字节码动态生成代理对象
  2. Cluster：通过负载均衡、容错策略选择一个 Provider 节点
  3. Protocol：决定使用 Dubbo 协议或 REST 协议进行通信
  4. Exchange：封装请求/响应模型，处理同步/异步转换
  5. Transport：基于 Netty 等通信框架进行网络传输
  6. Serialize：序列化请求对象为二进制流

四、通信协议与序列化

4.1 支持的协议

协议 特点 适用场景
dubbo 单一长连接，NIO 异步通信，适合小数据量高频调用 服务间常规调用，推荐
rmi Java 原生远程方法调用，阻塞 IO 少量与遗留系统集成
hessian 轻量级二进制协议，支持跨语言 跨语言调用场景
http 基于 HTTP 表单提交 需要穿透防火墙或跨平台调用
rest 基于 HTTP + JSON，无状态 对外开放 API 时使用

单连接模型：dubbo 协议默认使用单一长连接，服务提供者和消费者之间共用一个连接，通过多路复用并发处理请求，减少了连接数和握手开销。但也意味着大文件传输会阻塞其他请求。

4.2 序列化方式

  • Hessian2（默认）：跨语言，兼容性好，性能中等
  • Kryo：高性能，需要注册类，线程不安全（需配合 ThreadLocal）
  • FST：高性能，完全兼容 JDK 序列化协议
  • Protobuf：跨语言，压缩率高，需预先定义 IDL 文件

面试常问：为什么不用 Java 原生序列化？因为 Serializable 序列化流大、性能差、无法跨语言。

五、负载均衡策略

Dubbo 提供多种负载均衡策略，可通过 @Reference(loadbalance = "xxx") 配置。

5.1 随机 (Random，默认)

按权重设置随机概率，碰撞少，适合服务提供者能力均衡的场景。

// 配置
@Reference(loadbalance = "random")
private SomeService someService;

5.2 轮询 (RoundRobin)

按公约后的权重轮询分发请求，存在“累积请求”问题：慢的机器会积压更多请求。Dubbo 默认实现是加权轮询。

5.3 最少活跃调用数 (LeastActive)

选择活跃数（正在处理请求数）最少的机器。能智能地分配请求给当前负载最小的节点。

5.4 一致性哈希 (ConsistentHash)

相同参数的请求总是发送到同一 Provider，适合有状态场景（如缓存预热）。

权重配置：可在 Provider 的 @Service(weight = 200) 中配置，数值越大处理能力越强。

六、集群容错机制

cluster 属性决定调用失败时的处理策略。

模式 行为
Failover（默认） 失败自动切换，重试其他服务器，重试次数可配置（retries=2）
Failfast 快速失败，只调用一次，失败立即报错
Failsafe 出现异常时直接忽略（适合日志记录等非关键调用）
Failback 失败后定时重发，直到成功
Forking 并行调用多个服务器，只要一个成功即返回
Broadcast 广播调用所有提供者，任意一个出错则报错

配置示例：

@Reference(cluster = "failover", retries = 3)
private OrderService orderService;

注意：幂等性是重试的前提，非幂等写操作应谨慎配置重试。

七、服务注册与发现

Dubbo 支持多种注册中心，最常用的是 Zookeeper 和 Nacos。

7.1 Zookeeper 注册中心

采用树形结构存储服务信息 /dubbo/com.xxx.XxxService/providers/...。
优点：强一致性（CP），数据实时性高。
缺点：集群过半宕机时不可用，不适合大规模推送。

7.2 Nacos 注册中心

同时支持 AP（最终一致）和 CP，功能更丰富，集成了配置中心。
优点：高可用，支持服务健康检查、动态配置。
缺点：性能略低于 ZK。

常见问题：Dubbo 是如何感知服务下线的？
Zookeeper 通过临时节点，Provider 断开时 ZK 会话超时自动删除节点，Consumer 通过 Watcher 机制及时收到通知更新本地地址列表。

八、Dubbo 与 Spring Cloud 对比

维度 Dubbo Spring Cloud
通信协议 多协议，默认 Dubbo（单长连接+NIO） HTTP/REST (基于 Tomcat/Jetty)
注册中心 Zookeeper、Nacos、Redis、etc Eureka、Consul、Nacos、Zookeeper
服务调用方式 RPC（接口代理） RestTemplate / Feign（HTTP 客户端）
性能 更高（二进制、长连接、非阻塞） 相对较低（HTTP 短连接或长连接池）
服务治理 较丰富（路由、熔断、权重调整等） 需要集成 Sentinel、Hystrix 等
生态整合 专注 RPC，可集成 Spring 全家桶，集成配置中心、网关、消息等

在实际项目中，二者经常混合使用，如用 Dubbo 承担内部高性能 RPC 调用，用 Spring Cloud Gateway 作为外部 API 网关。

九、常见问题与优化

9.1 消费者调用超时

  • 检查服务端处理时间：是否慢 SQL、逻辑耗时长、GC 停顿
  • 调整超时时间：@Reference(timeout = 3000) 设置超时（毫秒）
  • 检查线程池：Dubbo 默认有业务线程池（200 个线程），若满则请求排队超时
  • 网络状况：跨机房时适当增大超时

9.2 生产者线程池耗尽

Dubbo Provider 默认线程池 fixed，大小 200。高并发下可能不够。

# 线程池配置
<dubbo:protocol name="dubbo" dispatcher="all" threadpool="cached" threads="500"/>
  • threadpool：推荐 cached（弹性）或 fixed（固定）
  • queues：队列大小，0 表示同步队列（无排队直接拒绝），可能导致大量拒绝异常

9.3 服务暴露失败

常见原因：

  • 注册中心连接失败（检查网络、端口、配置）
  • 端口冲突（dubbo 默认 20880，检查 netstat）
  • 未扫描到 Dubbo 注解（@EnableDubbo 或 XML 配置路径错误）

9.4 序列化失败

  • NotSerializableException：确保传递的对象实现 Serializable 接口
  • ClassNotFoundException：Consumer 和 Provider 的类版本不一致，尽量让接口定义独立成一个公共 jar

9.5 重试导致的数据重复

对于非幂等写操作，务必配置 retries = 0 或使用 cluster = "failfast"，并配合幂等性设计（如数据库唯一索引、token 机制）。

十、面试高频追问

  1. Dubbo 如何解决粘包/拆包？
    Dubbo 使用 Netty 作为传输层，通过自定义协议头（魔数、消息长度等）进行消息定界。

  2. 服务调用的最佳实践

    • 接口粒度适中，避免过大或过小
    • 传输对象尽量使用公共 Jar 包统一管理
    • 合理配置超时和重试
    • 依赖治理：使用 check=false 避免启动时强依赖不可用

  3. Dubbo 的 SPI 机制
    Dubbo 扩展点使用增强的 SPI，通过 META-INF/dubbo/ 下的配置文件加载扩展实现，实现了微内核 + 插件化架构，几乎所有组件都可替换。

  4. 如何实现灰度发布
    结合路由规则（ConditionRouter），根据参数或 IP 将部分流量导入灰度环境。

理解 Dubbo 的设计思想和常见配置，不仅能帮你通过面试，更能在日常开发中写出稳健的分布式系统代码。
