2024-05-03 基础知识 加密算法 0 评论

计算机中常见加密手段及其原理

本文不是工具使用手册，而是对加密手段背后数学原理与设计逻辑的梳理。
旨在回答两个问题：它解决了什么安全问题？它凭什么能解决？

1. 对称加密 (Symmetric Encryption)

1.1 解决什么问题

在不可信的信道上，让通信双方安全地传输消息，且只有他们能读懂。

1.2 核心原理

通信双方事先协商好同一个密钥 K。

  • 加密：密文 = Encrypt(明文, K)
  • 解密：明文 = Decrypt(密文, K)

对称加密的安全性完全依赖于密钥的保密性，而非算法本身的保密。现代密码学算法全部公开，经得起数学推敲。

1.3 代表算法

  • AES (Advanced Encryption Standard)：目前全球最主流的对称加密算法，分组加密，密钥长度 128/192/256 位。广泛应用于 HTTPS、文件加密等。
  • ChaCha20：流加密算法，移动端友好，TLS 1.3 推荐套件之一。

1.4 关键局限

密钥分发问题：通信双方如何在事先不通信的情况下，安全地协商出同一个密钥？这正是非对称加密要解决的。

2. 非对称加密 (Asymmetric Encryption)

2.1 解决什么问题

让从未见过面、没有共享秘密的两个人，也能在公开信道上安全通信。

2.2 核心原理

每个人拥有一对数学上关联、但无法相互推导的密钥：

  • **公钥 (Public Key)**：可以公开给任何人。
  • **私钥 (Private Key)**：必须绝对保密。

核心事实：

  • 用公钥加密的密文，只有对应的私钥能解开。
  • 用私钥加密的签名（实际是签名，见后文），只有对应的公钥能验证。

2.3 代表算法

  • RSA：基于大整数分解难题。两个大素数乘积容易，但对其结果进行分解极难。
  • ECC (Elliptic Curve Cryptography)：基于椭圆曲线离散对数难题。相比 RSA，在相同安全强度下密钥更短、计算更快，是现代首选。

2.4 典型应用

  • 密钥交换：用非对称加密来安全地协商一个临时对称密钥（如 HTTPS 握手），之后的数据用对称加密保护——因为非对称加密太慢，不适合大量数据。
  • 数字签名（见后文）。

3. 哈希 (Hash / 摘要)

3.1 解决什么问题

验证数据的完整性——保证消息在传输或存储后，没有被篡改（哪怕改了一个比特）。

3.2 核心原理

把一个任意长度的数据，映射成一个固定长度的指纹（摘要）。这个映射必须满足：

  • 单向性：从摘要无法反推出原文。
  • 抗碰撞性：找不到两个不同的输入，产生相同的摘要。
  • 雪崩效应：原文改一点，摘要面目全非。

3.3 代表算法

  • SHA-256 / SHA-3：目前安全标准，广泛应用于数字签名、区块链。
  • Blake2 / Blake3：性能更优的现代哈希函数。
  • ⚠️ MD5 / SHA-1：已被攻破（可构造碰撞），任何安全场景都不应继续使用。

3.4 自己不能解决什么

哈希仅能验证数据“是否被改过”，但无法验证“这个数据是谁发的”。因为任何人都可以计算一个数据的哈希值，然后把数据和哈希一起发过来。

4. 消息认证码 (MAC)

4.1 解决什么问题

同时验证完整性 + 来源真实性。即保证消息没被篡改，而且确实来自声称的那个人。

4.2 核心原理

通信双方共享一个密钥 K。发送方用 K 和消息一起计算出 MAC值，附在消息后发出；接收方用同样的 K 和收到的消息重新计算，若一致，则证明：

  • 消息未被篡改（篡改了 MAC 就对不上）。
  • 消息确实来自持有 K 的另一方。

4.3 代表算法

  • HMAC-SHA256：基于哈希的 MAC，最常见的实现。
  • CBC-MAC / CMAC：基于分组密码的 MAC。

4.4 致命局限

Bob 可以验证消息来自 Alice（因为只有他俩有共享密钥），但 Bob 无法向第三方证明这条消息是 Alice 发的。因为 Bob 自己也能生成一条一模一样的消息和 MAC。这就是 抗抵赖 的缺失，数字签名正是为此而生。

5. 数字签名 (Digital Signature)

5.1 解决什么问题

提供完整性、来源认证、不可抵赖三者合一。发送方不能否认自己发送过这条消息。

5.2 核心原理

结合哈希和非对称加密：

  1. 发送方对原文计算哈希（得到摘要）。
  2. 发送方用自己的私钥对摘要加密（即签名）。
  3. 接收方收到原文和签名后：
    • 对原文计算哈希得到 h1。
    • 用发送方的公钥解密签名，得到 h2。
    • 若 h1 == h2，则验证通过。

谁动过原文，哈希就变了，签名就失效。而签名只能用发送方的私钥生成，所以：

  • 接收方可以确信消息来源。
  • 第三方也可以验证（因为有公钥即可），实现了不可抵赖。

5.3 代表算法

  • RSA-PSS
  • ECDSA (基于椭圆曲线)
  • Ed25519 (基于 EdDSA，现代推荐)

5.4 关键前提（信任之锚）

公钥本身怎么确保是真的？如果攻击者把公钥替换成自己的，整个体系崩溃。于是需要公钥基础设施 (PKI) 和数字证书来建立信任链，那又是另一个话题了。

6. 一张总结表

技术手段 解决的问题 依赖的密钥 关键能力
对称加密 数据保密 双方共享同一个密钥 加解密快，密钥分发难
非对称加密 密钥协商、保密 公钥/私钥对 解决密钥分发，但速度慢
哈希 数据完整性 无 防篡改，但无身份信息
消息认证码 完整性 + 来源认证 共享密钥 防篡改+防伪造，但无法防止内部抵赖
数字签名 完整性 + 来源认证 + 抗抵赖 私钥签名、公钥验证 第三方可验证，责任不可否认

7. 扩展与思考

现代安全通信（如 HTTPS）通常是一套组合拳：

  1. 用非对称加密（如 ECDHE）安全地协商出一对临时对称密钥。
  2. 用对称加密（如 AES-GCM）加密实际数据流，并用 MAC 保证完整性。
  3. 用数字签名（证书链）验证服务器的公钥真实性。
