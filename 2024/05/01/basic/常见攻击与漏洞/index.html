常见攻击与漏洞

2024-05-01 基础知识 safe 0 评论

服务器一旦暴露在公网，就注定会成为攻击者的目标。无论是个人博客还是企业应用，了解常见的攻击手法并提前布防，是每一个开发者和运维人员的必修课。

1. DDoS 攻击（分布式拒绝服务）

攻击原理
攻击者控制大量“肉鸡”（被入侵的计算机）同时向目标服务器发起海量请求，耗尽服务器的带宽、CPU 或内存资源，导致正常用户无法访问。

常见类型

  • SYN Flood：伪造大量 TCP 连接请求，耗尽连接队列
  • UDP Flood：发送大量 UDP 包，占满带宽
  • HTTP Flood：模拟正常 HTTP 请求，打垮应用层

解决方案

  • 使用高防 IP 或 CDN：如 Cloudflare、阿里云 DDoS 高防，在流量进入服务器前清洗
  • 配置防火墙限流：使用 iptables 限制单个 IP 的并发连接数
  • 关闭无用端口：只保留 80、443 等必要端口，减少攻击面
  • 开启 SYN Cookie：echo 1 > /proc/sys/net/ipv4/tcp_syncookies
  • 应用层限流：在 Nginx 中配置 limit_req_zone 和 limit_conn，限制单个 IP 的请求速率

2. SQL 注入

攻击原理
攻击者在输入框（如登录表单、搜索框）中提交精心构造的 SQL 语句，如果后端没有对输入做过滤，这些恶意 SQL 就会被数据库执行，导致数据泄露、篡改甚至删除。

经典示例 ' OR '1'='1' --

解决方案

  • 使用参数化查询（预编译语句）：Java 中的 PreparedStatement、Go 的 database/sql 占位符，从根本杜绝拼接 SQL
  • ORM 框架：如 MyBatis 的 #{} 会进行参数化，避免使用 ${} 拼接
  • 输入校验：对用户输入进行类型、长度、格式的严格验证，过滤特殊字符
  • 最小权限原则：应用连接数据库的账号只授予必要权限，禁用 DROP、ALTER 等高危操作
  • **Web 应用防火墙 (WAF)**：识别并拦截 SQL 注入特征

3. XSS（跨站脚本攻击）

攻击原理
攻击者将恶意脚本注入网页中，当其他用户访问该页面时，脚本会在受害者浏览器中执行，从而窃取 Cookie、劫持会话或重定向到钓鱼网站。

分类

  • 反射型 XSS：恶意脚本通过 URL 参数传入，服务端直接回显
  • 存储型 XSS：恶意脚本被永久存储在数据库（如评论、文章），所有访问者都会被攻击
  • DOM 型 XSS：客户端 JavaScript 动态修改 DOM 时被利用

解决方案

  • 输出编码：在页面输出用户可控的内容时，进行 HTML 实体编码（如 < 转为 &lt;）
  • HttpOnly Cookie：服务端设置 Cookie 时加上 HttpOnly 属性，使 JavaScript 无法读取
  • **内容安全策略 (CSP)**：配置 HTTP 头 Content-Security-Policy，限制可执行的脚本来源
  • 输入过滤：对用户提交的数据进行白名单校验，去掉或转义 <script> 等标签

4. CSRF（跨站请求伪造）

攻击原理
攻击者诱导已登录的用户点击一个链接或加载一个图片，利用浏览器自动携带的 Cookie，向目标网站发送伪造的请求（如修改密码、转账等）。

解决方案

  • Anti-CSRF Token：表单中加入随机生成的 Token，服务端验证，攻击者无法伪造
  • SameSite Cookie：将 Cookie 的 SameSite 属性设为 Strict 或 Lax，限制第三方请求携带 Cookie
  • 验证 Referer/Origin 头：检查请求来源是否为本站域
  • 敏感操作二次验证：修改密码、支付等操作要求输入验证码或短信验证

5. SSH 暴力破解

攻击原理
攻击者通过自动化工具（如 Hydra、Medusa）反复尝试不同的用户名和密码组合，一旦试出弱口令，就能直接控制服务器。

解决方案

  • 修改默认 SSH 端口：将 22 改为高位端口（如 2222），减少自动扫描命中率
  • 禁用密码登录，只用密钥对：PasswordAuthentication no
  • 安装 fail2ban：自动识别并封禁多次失败登录的 IP
  • 限制登录 IP：通过 iptables 或安全组规则，只允许特定 IP 或 IP 段访问 SSH
  • 使用非 root 用户登录，需要提权时 sudo

6. 文件上传漏洞

攻击原理
应用允许用户上传文件时未做严格校验，攻击者可上传恶意脚本（如 PHP 一句话木马），然后通过访问文件路径在服务器上执行任意命令。

解决方案

  • 白名单校验扩展名：只允许 .jpg、.png、.pdf 等安全类型
  • 检查 MIME 类型和文件头：防止伪造扩展名
  • 重命名文件：使用随机字符串（如 UUID）作为文件名，不保留原始文件名
  • 将上传目录设在 Web 根目录外，不能直接通过 URL 访问
  • 限制上传目录的执行权限：关闭该目录的脚本执行功能

7. 中间人攻击 (MITM)

攻击原理
攻击者位于客户端与服务器之间，通过 ARP 欺骗、伪造 Wi-Fi 热点等方式截获或篡改通信内容，窃取敏感信息。

解决方案

  • 全站启用 HTTPS：申请免费 SSL 证书（如 Let’s Encrypt），并设置 HSTS（HTTP Strict Transport Security）
  • **证书固定 (Certificate Pinning)**：在客户端代码中预埋服务端证书的公钥，防止伪造证书
  • 避免使用公共 Wi-Fi 访问敏感服务，或始终开启 VPN
  • 使用 DNSSEC 和加密 DNS（如 DNS over HTTPS）

8. 恶意扫描与信息泄露

攻击原理
攻击者通过端口扫描、目录枚举等探测你的服务指纹（如 Web 服务器版本、开放端口），为后续精确攻击做准备。同时，不注意的信息泄露（如错误页面输出堆栈信息、.git 目录暴露）也极大降低攻击成本。

解决方案

  • 关闭多余端口，只保留业务必须的服务
  • 隐藏服务版本信息：Nginx 中 server_tokens off;，Tomcat 自定义错误页
  • 定期扫描敏感文件：.git、.env、备份文件等应全部删除或移出 Web 目录
  • 自定义 40x、50x 错误页面，避免暴露框架默认信息
  • 使用 WAF 配置规则拦截常见的扫描行为

总结：构建纵深防御体系

安全不是某个单一产品的事，而是一个多层次、持续改进的过程。记住下面这些原则：

  1. 最小权限：每个服务、每个账号只拥有能完成其工作的最小权限
  2. 纵深防御：网络层、系统层、应用层、数据层环环相扣，不依赖单一防线
  3. 默认拒绝：除非明确允许，否则一切访问默认拒绝
  4. 持续更新：及时关注 CVE 漏洞通告，定期升级系统和依赖
  5. 日志与监控：记录关键操作日志，部署入侵检测系统 (IDS)，发现异常立即告警

将这份清单逐一落实到你的服务器和应用中，就能有效抵御 90% 以上的常见攻击。安全建设永远在路上，愿你的服务坚如磐石。
