Zookeeper

用于协调分布式组件，包括配置管理、名字服务、分布式锁、集群管理。

一、基础概念

1.1 ZooKeeper 是什么？

ZooKeeper 是一个开源的分布式协调服务，为分布式应用提供一致性服务。它使用类似文件系统的树形数据结构（ZNode）存储数据，通过 ZAB（原子广播协议） 保证数据一致性和高可用。

1.2 核心角色

  • Leader：负责写请求的处理和事务提议的发起与协调，集群中只有一个
  • Follower：参与选举和投票，处理读请求，将写请求转发给 Leader
  • Observer：不参与投票，只处理读请求，用于扩展读性能，不降低写性能

二、数据模型

2.1 ZNode 节点

ZooKeeper 的数据模型类似一棵树，每个节点称为 ZNode，可以存储少量数据（默认上限 1MB）。

ZNode 分为：

  • 持久节点：创建后一直存在，直到显式删除
  • 临时节点：与客户端会话绑定，会话结束自动删除
  • 持久顺序节点：持久节点 + 单调递增序号
  • 临时顺序节点：临时节点 + 单调递增序号

2.2 节点属性

  • Stat：包含版本号（version）、ACL 信息、时间戳等，每次更新版本号递增
  • Watcher：客户端可以在节点上设置监听，当节点数据或子节点发生变化时，服务端会通知客户端（一次性触发）

三、核心机制

3.1 ZAB 协议（原子广播）

ZAB 是 ZooKeeper 的一致性协议，包含两种模式：

  • 崩溃恢复：集群启动或 Leader 宕机时进行 Leader 选举
  • 消息广播：Leader 将写请求以事务 Proposal 形式发送给所有 Follower，过半节点 ACK 后提交

3.2 Leader 选举

选举原则：Zxid（事务 ID）大的优先，若相同则 server id 大的优先。通过 FastLeaderElection 算法实现，快速选出新 Leader。

3.3 会话机制

客户端与服务器建立 TCP 连接后，创建 Session。Session 有超时时间，服务器在超时时间内未收到心跳，则标记会话过期并清除相关临时节点。

重要：临时节点的生命周期与会话绑定，可用于服务注册发现中的健康检测。

3.4 Watcher 机制

  • 客户端注册 Watcher 到某 ZNode
  • 当 ZNode 发生指定事件（如数据变更、节点删除等），服务端触发通知
  • 一次性：通知后 Watcher 失效，客户端需重新注册（防止丢失事件，可在回调中重新注册）

四、典型应用场景

4.1 分布式配置管理

将配置信息存储在 ZNode 中，客户端 watch 该节点，配置变更时实时通知客户端。

// 读取配置
byte[] configData = zk.getData("/config/app", true, null);
// 监听变化
zk.exists("/config/app", event -> {
    if (event.getType() == Watcher.Event.EventType.NodeDataChanged) {
        // 重新读取配置
    }
});

4.2 分布式锁

利用临时顺序节点实现公平锁：

  1. 所有竞争者创建一个临时顺序节点，如 /locks/lock-0000000001
  2. 判断自己是否是最小节点，若是则获取锁
  3. 若不是，则 watch 前一个节点
  4. 前一个节点释放锁（客户端断开或删除节点），当前节点被通知，重新判断

4.3 服务注册与发现（Dubbo 中常用）

  • Provider 创建临时节点 /dubbo/com.xxx.XxxService/providers/192.168.1.100:20880
  • Consumer watch /dubbo/com.xxx.XxxService/providers，有变化时拉取新列表

4.4 Master 选举

利用临时节点或临时顺序节点，多个节点尝试创建同一路径（如 /master），创建成功者为 Master，其余节点 watch 该节点，当 Master 宕机时重新选举。

五、集群部署与高可用

  • 集群节点数推荐奇数（3、5、7），因为 ZAB 需要过半存活才能工作

  • 所有节点数据一致，客户端连接任意节点均可

  • 常用配置：tickTime、initLimit、syncLimit、dataDir

    zoo.cfg 示例

    tickTime=2000
    initLimit=10
    syncLimit=5
    dataDir=/var/lib/zookeeper
    clientPort=2181
    server.1=192.168.1.1:2888:3888
    server.2=192.168.1.2:2888:3888
    server.3=192.168.1.3:2888:3888

六、常见问题与排查

6.1 集群启动失败或脑裂

  • 检查半数以上节点是否存活
  • 网络分区可能导致脑裂，但 ZAB 依靠过半确认可避免多主写入
  • 防火墙是否放行端口（2181、2888、3888）

6.2 客户端频繁超时

  • 增大 tickTime 和 sessionTimeout
  • 检查 JVM GC 是否频繁停顿（导致心跳无法发送）
  • 网络延迟高或负载过高

6.3 节点数据丢失

  • 确认节点类型：临时节点在会话过期后会被删除
  • 检查 dataDir 磁盘是否有空间，事务日志是否损坏
  • 是否误操作删除了节点

6.4 性能问题

  • 读压力大时可增加 Observer 节点分摊
  • 减少每次 Watcher 触发的业务逻辑，尽快重新注册
  • 单个节点数据不宜过大（不超过 1MB），否则影响网络传输和磁盘 I/O

七、面试高频问答

问：ZooKeeper 如何保证数据一致性？
答：通过 ZAB 协议实现原子广播，写请求必须由 Leader 发起提案，得到过半 Follower 确认后才提交，保证分布式环境下的强一致性。

问：ZooKeeper 的典型使用场景有哪些？
答：配置中心、服务注册与发现、分布式锁、Master 选举、分布式队列。

问：为什么 ZooKeeper 集群推荐奇数节点？
答：过半原则决定容错能力。例如：3 节点允许挂 1 台，4 节点同样允许挂 1 台，多 1 台浪费资源；奇数节点可避免平票。

问：ZooKeeper 和 Eureka / Nacos 的区别？
答：ZooKeeper 强调 CP（强一致），Eureka 强调 AP（高可用），Nacos 同时支持 CP 和 AP，且内置配置中心。

问：ZooKeeper 的 Watcher 为什么是一次性的？
答：避免服务端维护大量持续监听的开销；客户端在回调中重新注册即可保证后续通知不丢失。

ZooKeeper 虽简单，却撑起了海量分布式系统的协调重任，理解其原理对构建健壮的分布式架构至关重要。
