RAG 实战：LangChain / Spring AI + Elasticsearch 构建私有知识库问答系统

2026-02-28 系统设计 Elasticsearch, LangChain, RAG, Spring AI, 向量检索, 知识库 0 评论

去年给公司内部做了一套私有知识库问答系统，把 Elasticsearch 当检索底座，分别用 LangChain 和 Spring AI 各实现了一版。两个方案踩了一些坑，但也跑通了从文档上传到对话回复的完整链路。这篇文章就按项目推进的顺序，把流程、关键决策和常见的坑整理出来，供参考。

一、RAG 解决的到底是什么问题

大模型有两个明显的短板：训练数据有截止日期，遇到公司内部的专业内容容易编造事实。每次看到它信誓旦旦地给出错误答案，就知道不能只用模型本身，得给它配一本能实时查阅的“参考书”。

RAG 就是这本书的实现方式：先把私有文档向量化存起来，用户提问时检索最相关的内容，再把内容连同问题一起塞进 Prompt，让模型基于这些材料回答。整个体系分成两个阶段：

  • 离线阶段：把 PDF、Word、Markdown、数据库记录等文档解析成纯文本，再按固定长度切片，通过 Embedding 模型转成向量，存入 Elasticsearch。
  • 在线阶段：用户提问同样先转成向量，去 ES 里用 KNN 检索出最相似的 K 个片段，拼接到 Prompt 中，让 LLM 生成最终答案。

二、文档切分的坑比想象得多

把文档转成文本很简单，Apache Tika 或者 Python 的 pdfplumber、python-docx 随便一个库就能搞定。但到了分块（chunking）这一步，问题就来了。

最初直接用 LangChain 的 RecursiveCharacterTextSplitter，按 512 个字符切，结果切出来的片段经常一句话被拦腰截断，上下文支离破碎。后来改成了按“段落 + 重叠”的方式：先按自然段落切分，段落太长的再切，最后每个 chunk 都带上前后各 50 个 token 的重叠，效果好了很多。

还有一些特殊文档，比如技术手册里代码块和表格非常多，按行切容易把这些东西打散。最后给这类文档写了专门的解析器，代码块保持完整，表格转成 Markdown 格式再切片。

分块大小没有银弹。经验是：对于普通技术文档，512 token 左右够用；对于法律合同这种长句密集的，可以放大到 1024 token。但越大越容易混入无关信息，检索准确度会下降。

三、向量化和检索：从“能用”到“好用”的调整

Embedding 模型选的是 text-embedding-3-small（用在线 API）和 bge-large-zh-v1.5（本地部署），维度都是 768。ES 里建索引的 mapping 大概是：

PUT /knowledge_base
{
  "mappings": {
    "properties": {
      "content": { "type": "text", "analyzer": "ik_max_word" },
      "content_vector": {
        "type": "dense_vector",
        "dims": 768,
        "index": true,
        "similarity": "cosine"
      },
      "title": { "type": "text" },
      "category": { "type": "keyword" },
      "source_file": { "type": "keyword" }
    }
  }
}

写入时通过 _bulk 批量灌，单个文档带上 content_vector 字段。ES 8 以后 HNSW 的索引速度和召回率都挺好，没必要自己再套一层 FAISS。

检索阶段一开始只用了纯向量 KNN：

GET /knowledge_base/_search
{
  "knn": {
    "field": "content_vector",
    "query_vector": [ ... ],
    "k": 5,
    "num_candidates": 20
  }
}

但业务方反馈“有时候关键词特别明确的查询，答案反而不准”。一看日志，全是“合同编号 xxx 的金额是多少”这种问题，KNN 检索可能因为语义相似度把其他合同的金额也拉进来了。

解决方法是在 ES 里加上混合检索：先用 multi_match 在 title 和 content 上做关键词匹配，再用 KNN 向量检索，把两个结果集用倒数排名融合（Reciprocal Rank Fusion, RRF）合并后取 Top-K。调整后准确率提升明显。虽然多了一步，但查询延迟在几十毫秒以内，完全能接受。

四、两个版本：LangChain 和 Spring AI 分别怎么搭

Python 版用的是 LangChain，整体链式调用很顺。从 ES 检索回来，把文档内容拼成 Prompt 模板：

from langchain.prompts import ChatPromptTemplate
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个公司内部知识库助手，请根据以下资料回答问题。如果资料中找不到答案，请说不知道。"),
    ("user", "资料：\n{context}\n\n问题：{question}")
])

然后调用 LLM，配合 StrOutputParser 输出。LangChain 的好处是生态齐全，文档加载器、文本分割器、Retriever 接口一应俱全，快速出原型很快。

但公司后端技术栈是 Java，部署 Python 服务增加了运维成本。于是又用 Spring AI 搞了一版。核心用的是 ElasticsearchVectorStore 和 OpenAiChatModel：

@Bean
public VectorStore vectorStore(ElasticsearchClient esClient, EmbeddingModel embedding) {
    return new ElasticsearchVectorStore(
        new ElasticsearchVectorStoreOptions("knowledge_base"),
        esClient,
        embedding
    );
}

@Bean
public OpenAiChatModel chatModel() {
    return OpenAiChatModel.builder()
        .apiKey(System.getenv("OPENAI_API_KEY"))
        .model("gpt-4o-mini")
        .build();
}

查询时直接用 VectorStore.similaritySearch(query, 5)，然后把结果传给 ChatModel。Spring AI 目前还在快速迭代，很多接口不稳定，之前版本升级就遇到 ElasticsearchVectorStore 构造函数变更，改了不少代码。但好处是可以无缝集成 Spring Boot 生态，和现有的权限、日志、监控对接很方便。

两个版本比较：

  • 快速验证、数据科学家做实验，用 LangChain。
  • 生产环境且技术栈是 Java，用 Spring AI 更省心，虽然版本还不算稳定，但架构一致。

五、效果和没解决的尾巴

上线的知识库覆盖了公司两个部门的技术文档和规章制度，问答准确率在 85% 左右。几个明显的问题还在解决：

  • 用户问“怎么 A”时，如果文档里同时有“A 的介绍”和“A 的配置说明”，检索可能混进不相关的内容。后续打算加 metadata 过滤，让用户在提问时先选分类。
  • 多轮对话现在每轮都独立检索，没有利用历史信息。下一步要加入查询重写（query rewriting），把对话上下文融入检索。
  • 有些带表格的问题，模型直接引用表格文本会丢失结构，打算在分块阶段对表格做结构化提取，生成描述性文本。

整体来看，RAG 落地并不神秘，关键在于把每个环节的细节调好：分块策略、检索方式、Prompt 设计，各个点上差一点，最终效果就差很多。希望这些经验对正在做的开发者有所帮助。
